## 1. システムの概要
* **目的**: 
  指定された「求人ページのURL」から自動でテキストや給与などの情報を抽出し、事前に定義したルールブック（`rulebook.md`）とベクトル検索技術・OpenAI等の生成AIを用いて「原稿の内容に問題がないか」を自動審査するシステムです。
* **主な対象ユーザー**: 
  原稿審査を担当する社内メンバー。ブラウザからアクセスして利用します。

## 2. アプリケーション構成（ソフトウェア側）
アプリケーション本体は、手軽にWebサイトを作れる「Python 3」と「Flask」という組み合わせで作られています。

### コア機能と画面
1. **ユーザー認証（ログイン・登録）**
   * 個別のアカウントを持たせることで、誰が審査を依頼したかを記録します。
2. **AI審査画面 (`/` または `/review`)**
   * 審査したい求人のURLを入力するメイン画面です。
   * 入力後に裏側で「URL先からの文字の抽出（スクレイピング）」→「抽出ルールとの比較（ベクトル化）」→「AIによる判定文の作成」を順に行い、結果を画面に綺麗に表示します。
3. **審査履歴・検索画面 (`/history`)**
   * これまでに全ユーザーが依頼した審査の結果を一覧で振り返ることができます。
   * 履歴が増えても探しやすいように、画面上部から「企業名」や「審査依頼したユーザー名」でキーワード検索ができます。
   * **履歴の詳細表示**: AIの判定文だけでなく、審査する際に求人サイトから裏側で抜き出した文字データ（給与・応募資格・抽出した全文など）も一緒にアコーディオン形式（開閉メニュー）で保存・表示されます。

### データの保存方法と文字化け対策
* データベース（DB）へ履歴やユーザー情報を保存する際、本番環境の SQL Server では「英語のみのデータ」と「多言語（日本語）のデータ」の箱の形が厳格に区別されています。
* 日本語が `???` と文字化けするのを防ぐため、プログラム（`app.py`）内では専用の多言語文字列型（`NVARCHAR` / `db.Unicode`）を使って保存するように設計されています。
* また、求人タイトルなどが **200文字** を超えてDBの容量上限エラーにならないよう、保存する直前に安全に末尾を切り落とす安全装置（Truncation）も組み込まれています。

## 3. インフラ・サーバー構成（Azureクラウド側）
本システムはMicrosoft Azure上の各種サービスを組み合わせて稼働しています。

### ① Webサーバー: Azure App Service (Linux)
* **役割**: アプリの本体（Pythonプログラム）が24時間動いている場所です。
* **App Service プラン（価格帯）の選定制限について**:
  * AI（Azure OpenAI等）を利用した原稿審査のテキスト解析処理は、裏側で比較的重い処理が走ります。
  * **Free (F1) や Shared (D1) プラン**を使用した場合、1日あたりのCPU利用時間上限（クォータ）に数回の審査ですぐに達してしまい、アプリ自体が「クォータ超過」エラーとして一時停止してしまいます。
  * 審査業務を止めずに安定稼働させるためには、一日のCPU利用制限がない **Basic (B1) 以上のプラン（月額固定）** を選択することを強く推奨します。
* **Docker（コンテナ技術）の利用**:
  アプリを動かすための「専用の箱（コンテナ環境）」を作って動かしています（Debian 12ベース）。
  プログラムの設計図である `Dockerfile` の中には、ただPythonを動かすだけでなく、**「データベース（Azure SQL）とお話しするための翻訳ドライバー（Microsoft ODBC Driver 18）」**を箱の中にインストールする命令が書かれているため、Azureにデプロイするだけで最適なサーバー環境が自動で完成します。

### ② データベース: Azure SQL Database
* **役割**: ユーザーのログインIDやパスワード、過去の審査履歴データを永続的に安全に保管する「金庫」です。
  （※開発中のPC環境ではお手軽なローカルの `SQLite` を使用し、本番のAzure環境では規模の大きい `Azure SQL` に自動で繋がるように作られています）
* **セキュリティ（ファイアウォール）**:
  非常に硬いセキュリティ設定があり、**初期状態では外部（インターネット上のすべてのPC等）からの「直接アクセス」はすべて遮断**されます。
  * **★本アプリケーション (App Service) の利用について（全社員利用可能）**:
    App Serviceからの通信が許可されるよう設定（Azureサービスの許可・ポリシーの`Proxy`化）を行っているため、**利用者は「会社のIP」や「場所」関係なく、ブラウザからこのアプリのURLへアクセスして全員利用可能です。**（利用者はアプリへアクセスし、アプリが代表して内側からDBに書き込む仕組みです）
  * **★メンテナンス等で社内PCから手動で「DBの中身を直接見たい（クエリエディタ等）」場合のみ**:
    Webアプリを通さずにAzureポータルから直接SQLサーバーを覗き見ようとする場合に限り、その作業をしている人のご自身のPCのIPアドレスを許可リストに追加する必要があります。

## 4. 環境変数と秘密の設定
アプリが動作するために必要な「パスワード」や「鍵」は、ソースコード（GitHub）の中に直接書くと盗まれる危険があります。そのため、**環境変数** と呼ばれる金庫のような場所に保管しています。

本番環境（Azure）では **App Service の設定メニュー内にある「環境変数」画面** に以下の値を設定して運用します。
* `SECRET_KEY`: ログインの暗号化などに使う適当で複雑な文字列
* `OPENAI_API_KEY`: AIに審査を依頼するための認証キー `sk-....`
* `DATABASE_URL`: データベース（Azure SQL）に繋ぐための宛先とパスワードが合体したURL
  * ※ 例: `mssql+pyodbc://[ユーザー名]@[DBサーバー名]:[パスワード]@[DBサーバー名].database.windows.net/[DB名]?driver=ODBC+Driver+18+for+SQL+Server`

## 5. デプロイ（本番環境の更新）の仕組み
プログラムコードを修正して、本番（Azure）へ更新を反映する流れです。

1. **GitHub へのプッシュ（アップロード）**:
   開発者がパソコンでコードを書き換え、GitHub上の「デプロイ用ブランチ」へデータを送ります。
2. **自動デプロイ（反映）**:
   GitHub Actions または Azure の自動同期機能がコードの変更を検知し、Azure App Service に「新しいDockerの箱を作り直して起動してね」と命令を出します。
3. **App Serviceの再起動**:
   数分後、自動的にWebサーバーが再起動し、最新のコードがインターネット上の画面に反映されます。

> **【運用トラブル時の注意⚠️】**
> プログラムの改修に伴い、大規模な機能追加などにより「データベースのテーブルを作り直した場合（DROP TABLE等を行った場合）」、App Service（アプリ）側に向かって **必ずAzureポータル画面から手動で「再起動（Restart）」ボタンを押す必要があります**。そうしないと、アプリが消えた古いテーブルを探し続けて `Internal Server Error` になってしまうことがあります。
